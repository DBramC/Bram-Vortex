# Bram Vortex 🌀

Bram Vortex is an advanced, AI-powered Cloud Infrastructure Automation Platform, developed as the core deliverable of my **Bachelor's Thesis**.

Designed to bridge the gap between application development and cloud operations, Bram Vortex is a web-based system that intelligently analyzes application repositories to automatically determine, provision, and manage the most suitable cloud infrastructure. By leveraging a modern GitOps philosophy alongside AI agents, the platform enables automated, scalable, and highly secure deployment pipelines.

## 🧠 The AI Generation Engine 🤖

Bram Vortex doesn't just "guess"; it engineers. The platform utilizes a specialized multi-agent orchestration to transform a simple repository into a complex cloud environment:

* 🏗️ **Terraform Generator**: Architecting the backbone. It generates declarative HCL code to provision VPCs, Subnets, IAM roles, and Managed Services (ECS/EKS/EC2) across AWS, GCP, or Azure.

* 🛠️ **Ansible Configuration Engine**: For IaaS-heavy deployments, this service automates the "inside" of the VMs. It generates optimized **Ansible Playbooks** to handle OS hardening, dependency installation (Docker/Run-times), and container orchestration on raw compute instances.

* 🚀 **Pipeline Generator**: The glue of Continuous Delivery. It automatically crafts production-ready **GitHub Actions workflows**. These pipelines bridge the gap between infrastructure and code, automating builds, dockerization, and the final "push" to the newly created environment.

* ⚖️ **Architecture Validator (The Guardian)**: The platform’s safety net. Every AI-generated draft is cross-referenced by this service. It validates the code against real-world cloud constraints and security best practices, ensuring that the final package is not only syntactically correct but **production-ready and secure**.

## Cloud-Native Infrastructure & Kubernetes ☸️

The entire ecosystem is designed to run exclusively on **Kubernetes (K8s)**, following fully declarative configuration patterns:

* 🌉 **API Gateway / Ingress**: External traffic and routing between microservices are handled by the **Kong API Gateway**, which provides centralized control, TLS termination, and service load balancing within the cluster.
* 🗄️ **Stateful Data Management**: **PostgreSQL** 🐘 is used for data persistence. To ensure high availability and reliability in K8s, the database is provisioned as a **StatefulSet**, accompanied by separate **Persistent Volume Claims (PVCs)** for secure disk storage management.

---

## Zero-Trust Security & Secrets Management 🔐

Sensitive data 🛡️ (such as API keys for Google GenAI, DB credentials, and OAuth secrets) are not stored in images or plain text manifests. The system integrates enterprise-grade secrets management:

* 🏦 **HashiCorp Vault**: Utilizes Vault in combination with the **Secrets Store CSI Driver** of Kubernetes.
* 🔑 **SecretProviderClass**: Each microservice has its own definition, allowing K8s to pull encrypted secrets from Vault and mount them directly to the pod's file system at runtime.
* 🎫 **Stateless Authentication**: System security is reinforced using **JWT (JSON Web Tokens)**, generated by the Authentication Service and verified by every microservice for secure, stateless authorization.

## CI/CD & GitOps 🔄

The deployment pipeline is strictly structured around GitOps principles:

* 🤖 **Continuous Integration (CI)**: Pipelines orchestrated via **GitHub Actions**, validating builds and generating new container images.
* 🐙 **Continuous Deployment (CD)**: Managed by **Argo CD**, which constantly polls the repositories and syncs the K8s state according to the declarative configuration.

---

## Frontend 💻

The UI is a fully decoupled **React/TypeScript Single Page Application (SPA)** ⚛️. Deployed as a stateless container in Kubernetes, it communicates with backend services exclusively through RESTful API calls routed via the Kong Gateway. 🌐
